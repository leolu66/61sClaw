## 高级概述

OpenClaw 是一个个人 AI 助手平台，运行在你的笔记本电脑、VPS、衣橱里的 Mac Mini 或云容器上。它将 AI 模型和工具与你已经使用的消息应用连接起来：WhatsApp、Telegram、Discord、Slack、Signal、iMessage、Microsoft Teams 等。

OpenClaw 将你的 AI 助手视为基础设施问题，而不仅仅是提示音工程问题。OpenClaw 不试图让 LLM 通过巧妙的提示"记住"上下文或安全行为，而是围绕模型构建结构化的执行环境，包含适当的会话管理、内存系统、工具沙箱和消息路由。LLM 提供情报；OpenClaw 提供了操作系统。

你可以控制助手运行的地点、如何路由消息、可以使用哪些工具，以及会话的隔离方式。模型 API 调用仍然会发送到 Anthropic、OpenAI 或你的模型所在地；但对话历史、工具执行、会话状态以及所有编排逻辑都会留在你的基础设施中。

OpenClaw 面向开发者和高级用户，他们希望在任何消息应用都能访问个人 AI 助手，而不必将整个体验交给托管的第三方助手。如果你曾希望 Claude 或 GPT 能在你的 WhatsApp 私信、Slack 频道和 iMessage 聊天串中使用，同时在自己的硬件上运行智能，OpenClaw 就能满足你的体验。

OpenClaw 并不是围绕 AI 模型 API 的聊天机器人。它是为人工智能代理设计的操作系统。OpenClaw 将 AI 视为基础设施问题：会话、内存、工具沙箱、访问控制和编排。

### 工作原理

AI 模型提供智能；OpenClaw 提供了执行环境。

OpenClaw 采用以单一网关为中心的枢纽辐射架构，作为用户输入（WhatsApp、iMessage、Slack、macOS 应用、Web UI、CLI）与 AI 代理之间的控制平面：

- **网关**是一个 WebSocket 服务器，连接消息平台和控制接口，将每个路由消息分派到代理运行时。
- **代理运行时**端到端运行 AI 循环，从会话历史和内存中组装上下文，调用模型，执行工具调用，针对系统可用能力（浏览器自动化、文件操作、Canvas、调度作业等）执行，并持久化更新状态。

关键洞见是，OpenClaw 将接口层（消息来源）与助手运行时（智能和执行所在）分开。这意味着你可以通过任何已有的消息应用访问一个持久助手，对话状态和工具访问则集中管理在你的硬件上。

## 通过插件实现扩展性

OpenClaw 设计为在不修改核心代码的情况下进行扩展。插件通过四种主要方式扩展系统：

1. **频道插件**：额外的消息平台（Microsoft Teams、Matrix、Mattermost 等）
2. **内存插件**：替代存储后端（向量存储、知识图与默认 SQLite）
3. **工具插件**：自定义功能，超越内置的 bash、浏览器和文件操作
4. **提供者插件**：定制 LLM 提供者或自托管模型

插件系统存在于 `extensions/` 中，遵循基于发现的模型。`src/plugins/loader.ts` 中的插件加载器会扫描工作区包中的 `openclaw.extensions` 字段，验证已声明的模式，并在配置存在时进行热加载。

## 核心组成部分

### 1. 通道适配器

每个消息平台都有自己专用的适配器。有些适配器是内置的（你可以在 `src/telegram/`、`src/discord/`、`src/slack/`、`src/imessage` 等目录中找到它们），还有些可以通过通道插件添加。适配器实现了相同的接口，并规范了进出消息，使 OpenClaw 其他部分无需担心平台特定的怪癖。

每个适配器都实现了一个具有四大关键职责的通用接口：

#### 1.1 认证

认证因平台而异：
- **WhatsApp** 通过 Baileys 库使用二维码配对，将凭证存储在 `~/.openclaw/credentials` 中
- **Telegram 和 Discord** 使用通过环境变量 `TELEGRAM_BOT_TOKEN` 和 `DISCORD_BOT_TOKEN` 提供的机器人令牌
- **iMessage** 需要原生 macOS 集成，并且需要一个正确签名的 Messages 应用

#### 1.2 入站消息解析

入站消息解析处理不同平台数据格式带来的复杂现实。每个适配器负责提取文本，处理媒体附件（图片、音频、视频、文档），处理反应和表情符号，并维护线程或回复上下文。这个归一化层意味着 OpenClaw 的其他部分不需要知道消息是来自 WhatsApp 还是 Discord。

#### 1.3 访问控制

访问控制是在频道层面进行安全保护的地方。允许列表规定哪些电话号码或用户名可以与你的机器人互动；例如，`channels.whatsapp.allowFrom` 接受一系列电话号码。私信策略控制机器人如何处理来自未知发件人的私信。默认的"配对"策略要求在处理消息前获得批准。你可以设置为"开放"以接受所有私信（不推荐），或者设置为"禁用"以完全拒绝。组策略增加了另一层，包括提及要求和组特定的允许列表。

#### 1.4 外发消息格式化

每个平台都有自己的 markdown 方言、消息大小限制和媒体上传 API。适配器处理所有这些，包括将长消息分块以尊重平台限制、适当渲染标记、上传媒体文件，甚至管理输入指示和在场信息。

---

### 2. 控制接口

OpenClaw 提供了多种交互方式，每种方式都适合不同的使用场景和偏好：

#### 2.1 网页用户界面

Web UI 由基于 Lit 的网页组件构建，直接由网关提供服务。默认将浏览器指向 `http://127.0.0.1:18789/`，你会看到一个用于聊天、配置管理、会话检查、节点管理和健康监测的仪表盘。不需要单独的 Web 服务器；Gateway 会处理所有事情。

#### 2.2 CLI

CLI 通过 Commander.js 实现，从 `openclaw.mjs` 开始，流向 `src/cli/program.ts`，给你所有内容的命令行控制。用 `openclaw gateway` 启动网关。直接用 `openclaw agent` 调用代理。使用 `openclaw channels login` 配对 WhatsApp 或 Signal。通过 `openclaw message send` 程序发送消息。用 `openclaw doctor` 做健康诊断。或者带着 `openclaw` 一起走进引导设置。

#### 2.3 macOS 应用

macOS 应用采取了不同的方法。它用 Swift 编写，存在于 `apps/macOS` 中，它存在于你的菜单栏中，提供网关生命周期管理——启动、停止、重启，触手可及。它包含语音唤醒功能和按键通话叠加，将 WebChat 嵌入原生浏览器视图，甚至可以通过 SSH 控制远程网关。

#### 2.4 移动端

iOS 和 Android 的移动节点通过在连接握手中共角色："节点"来连接到网关，作为 WebSocket 节点。这不仅仅是聊天；这些应用展示了设备特定的功能，如摄像头访问、屏幕录制、定位服务和 Canvas 渲染。网关可以通过 `node.invoke` 协议方法调用这些功能，将你的手机变成代理工具的延伸。

### 3. 网关控制平面

网关存在于 `src/gateway/server.ts`，运行在 Node.js 22 或更新版本上。它是用 ws WebSocket 库构建的，默认绑定到 `127.0.0.1:18789`，仅支持环回以保证安全。这不仅仅是一台路由器；它是整个 OpenClaw 系统的唯一真相来源。

所有消息平台（通过 Baileys 库的 WhatsApp、通过 GrammY 的 Telegram、使用 discord.js 的 Discord）都通过这个中心点连接。CLI 工具、网页界面和移动应用都作为 WebSocket 客户端连接。当入站消息到达时，网关会将其路由到访问控制检查，解析哪个会话应处理该消息，并将其分发给相应的代理。它协调系统状态，包括会话、存在指标、健康监测和 cron 作业。更关键的是，它通过令牌或密码认证对任何非环回绑定进行安全，并为私信实现配对系统。

**设计原则**：

1. **每个主机只有一个网关**：这防止了 WhatsApp 会话冲突，因为 WhatsApp 的协议严格支持单设备
2. **整个协议是类型化的**：所有 WebSocket 帧都经过 JSON Schema 验证，而 JSON 模式本身是由 TypeBox 定义生成的
3. **系统是事件驱动的**：客户端订阅了代理、在场、健康和 tick 等活动，而不是不断问"有什么新消息？"
4. **任何副作用操作都需要幂等性密钥**：使重试逻辑安全并防止重复操作

本地连接（环回或同一尾网）可以自动批准，而远程连接则需要挑战响应签名和显式批准。

### 4. 代理运行时

代理运行时（在 `src/agents/piembeddedrunner.ts` 中实现）是 AI 交互实际发生的地方。它使用 Pi Agent Core 库（`@mariozechner/pi-agent-core`），采用 RPC 风格的调用模型，支持流响应。

从高层次来看，运行时每回合做四件事：
1. 解析会话
2. 组装上下文
3. 在执行工具调用时流式传输模型响应
4. 将更新状态持久化回磁盘

#### 4.1 会话解析

当消息到达时，运行时首先确定哪个会话应处理该消息：
- 你的私信会映射到主会话
- DM 通过频道映射到 `dm:<channel>:<id>`
- 群聊映射到 `group:<channel>:<id>`

会话不仅仅是身份识别——它们是安全边界。每种会话类型可能携带不同的权限和沙箱规则（例如：主会话可以在主机上运行工具，而 DM/组会话默认使用更严格的允许列表和 Docker 隔离）。

#### 4.2 上下文组装

会话解析完成后，运行时会为模型组装上下文。通常会：
- 从持久化的 JSON 会话文件加载会话历史（以确保每个会话保持连续性）
- 通过读取工作区配置文件并将其组合成单一指令栈，构建动态系统提示符
- 通过语义搜索（例如先前相关的转折或笔记）拉取内存，使模型只看到最相关的历史背景，而非不断增长的转录记录

这些组装好的上下文随后会被流式传输到配置好的模型提供者（Anthropic Claude、OpenAI GPT、Google Gemini，甚至本地模型），这样你就能逐个令牌获得响应，而不是等待最终的单个 blob。

#### 4.3 执行循环

当模型响应时，运行时会监控工具调用并拦截它们。如果模型请求一个工具（例如，运行 bash 命令、读写文件、打开浏览器并抓取网页），运行时会执行该工具，可能在 Docker 沙箱中执行，具体取决于会话的沙箱策略。每个工具结果都会被流回到持续的模型生成中，并被纳入并持续进行。会话回合结束后，运行时会将更新后的会话状态（消息、工具调用/结果及其他跟踪状态）持久化回磁盘。

#### 4.4 系统提示符架构

OpenClaw 通过组合多个来源来构建提示：

**工作区配置文件**：
- **AGENTS.md** — 核心代理指令（默认捆绑）。操作基准：代理允许做什么、全局约束以及适用于所有会话的不可协商规则
- **SOUL.md** — 性格与语气指导（可选）。语音和互动风格：代理如何说话和构建回答，但不包括工具行为或安全边界
- **TOOLS.md** — 用户专用工具约定（可选）。你个人关于工具在你环境中如何使用的笔记，而不是工具登记册

**动态上下文（每回合组装）**：
- **会话历史** — 当前对话中的近期消息
- **技能**（`skills/<skill>/SKILL.md`）— 技能定义和使用说明

**工具定义（自动生成）**：
- **内置工具**（`src/agents/pi-tools.ts`、`src/agents/openclaw-tools.ts`）— bash、浏览器、文件操作、画布及核心功能
- **插件工具**（注册于 `api.registerTool(toolName, toolDefinition)`）— 通过扩展系统添加的自定义工具

**基础系统**：
- **Pi Agent Core** — 来自代理运行库的基础指令

技能发现与技能注入是一个重要细节：OpenClaw 可以在运行时发现技能，但不会盲目地把所有技能注入每个提示。相反，运行时会选择性地只注入与当前回合相关的技能，以避免提示符膨胀并降低模型性能。

## 互动与协调

### 1. 画布与代理到界面（A2UI）

Canvas 是一个由代理驱动的可视化工作区，作为独立的服务器进程运行，默认为端口 18793。这种与主网关的分离提供了隔离（如果 Canvas 崩溃，网关仍能正常运行），并建立了不同的安全边界，因为 Canvas 提供的是代理可写内容。

流程是这样的：代理调用 canvas 更新方法，Canvas 服务器接收 HTML 并解析嵌入其中的任何 A2UI 属性，服务器通过 WebSocket 将这些内容推送给连接的浏览器客户端，客户端将 HTML 作为交互界面渲染。

**A2UI** 代表 Agent-to-UI，提供了一个声明式框架，代理生成带有特殊属性的 HTML。这些属性创建交互元素，无需代理编写 JavaScript。

当用户点击按钮时，客户端会向 Canvas 服务器发送一个动作事件，Canvas 服务器将其作为工具调用转发给代理。代理处理该动作，可能将任务标记为内部状态已完成，并调用带有新状态的画布更新。服务器将此更新推送给客户端，显示会自动刷新。

Canvas 支持跨多个平台。macOS 应用使用原生 WebKit 视图进行渲染。iOS 应用将 Canvas 包裹在 Swift UI 组件中。Android 使用 WebView 进行显示。当然，网页界面也可以直接在浏览器标签页中打开 Canvas。

### 2. 语音唤醒与通话模式

语音唤醒支持 macOS、iOS 和 Android 平台，提供始终在线的唤醒词检测。说"嘿，OpenClaw"，系统就会启动，准备听你的指令。或者，使用带有键盘快捷键的按键通话。音频会流向 ElevenLabs 进行转录，代理处理您的请求，回复通过 ElevenLabs 的文本转语音播放。

对话模式将这种模式扩展为持续对话。你可以与客服人员进行免提的来回对话，并带有中断检测功能，以便在客服说话时能及时介入。根据你的偏好配置自定义唤醒短语。

### 3. 多智能体路由

多智能体路由允许您将不同的信道或组导向完全隔离的代理实例。每个实例都可以有自己的工作区、模型和行为。

这种路由支持了多种强大的应用场景：
- 你可以为每个社区创建独立的人物角色，每个角色都针对该社区的文化和需求进行优化
- 不同情境可能有不同的工具访问
- 为不受信任的用户设立隔离沙盒
- 你可以在一个情境下测试新的代理行为，而不会影响其他情境中已建立且正常工作的代理

### 4. 会话工具（代理间通信）

会话工具使代理能够跨越不同会话进行协调，本质上提供了代理间的通信。当你希望客服人员协作处理复杂任务或共享信息，而无需在不同聊天场景间手动复制粘贴时，这些工具尤其有用。

- **sessions_list** — 发现正在进行的会话
- **sessions_send** — 发送另一个会话的消息
- **sessions_history** — 获取其他会话的文字记录
- **sessions_spawn** — 通过程序化创建新的会话以委派工作

### 5. 调度动作（Cron 作业）和外部触发器（Webhooks）

Cron 作业允许你安排代理动作在特定时间执行。想要每日总结吗？配置一个 cron 作业，每天早上 9 点触发，向主会话发送消息。Webhook 为代理动作提供了外部触发点。经典例子是电子邮件集成，Gmail 发布到一个 webhook 端点，触发代理动作。

## 深度分析：端到端消息流

让我们追踪一下当你向你的 OpenClaw 助手发送 WhatsApp 消息时会发生什么。理解这种流程可以揭示所有组成部分如何协同工作。

### 第一阶段：摄入

首先，Baileys 库会收到来自 WhatsApp 服务器的 WebSocket 事件。`src/whatsapp/` 中的 WhatsApp 适配器会解析该事件，提取消息文本、任何媒体附件以及关于发件人的元数据。

### 第二阶段：访问控制与路由

在消息继续前，它会到达访问控制层。这个发件人在你的允许列表里吗？如果是第一次私信，配对是否已经获批？如果任一检查失败，信息就到此为止。

假设访问控制通过，`src/auto-reply/reply.ts` 中的自动回复系统接管。它解析了哪个会话应处理此消息。如果是直接来自你，那就是主会话，功能完整。通过 WhatsApp 私信变成 `agent:main:whatsapp:dm:+123...` 了。群聊变成 `agent:main:whatsapp:group:120...@g.us` 了。每种会话类型都包含不同的权限和沙箱规则。

### 第三阶段：上下文组装

Agent Runtime 的 PiEmbeddedRunner 从磁盘加载已解析的会话。它通过读取工作区中的 AGENTS.md、SOUL.md 和 TOOLS.md，注入相关技能，并查询记忆搜索系统中语义相似的过去对话，以提供有用的上下文来组装系统提示。

这些丰富的上下文会被打包并流向你配置的模型提供者。

### 第四阶段：模型召唤

当模型响应时，运行时会监控工具调用。如果模型决定需要运行 bash 命令，运行时会拦截该调用并执行，如果这是非主会话，可能会在 Docker 沙箱中执行。如果模型想打开浏览器并抓取网站，运行时会启动 Chrome DevTools 协议自动化的 Chromium。每个工具结果都会被流回模型，模型将其纳入其持续响应中。

### 第五阶段：工具执行

（同上）

### 第六阶段：响应交付

响应块在到达时会通过网关返回。WhatsApp 适配器会格式化每个区块，将 markdown 转换为 WhatsApp 的标记格式，并尊重消息大小限制。格式化的消息会通过 Baileys 发送到 WhatsApp 服务器，最终传到你的手机。最后，运行时会将整个对话状态（你的消息、模型响应、所有工具调用和结果）持久化回磁盘上的会话 JSON 文件。

**延迟预算**：
- 访问控制：不到 10 毫秒
- 从磁盘加载会话：不到 50 毫秒
- 组装系统提示符：不到 100 毫秒
- 从模型中取回第一个令牌：200 到 500 毫秒（取决于网络状况）
- 工具执行：bash 命令通常在 100 毫秒内完成，浏览器自动化可能需要 1 到 3 秒

## 数据存储与状态管理

OpenClaw 会在你的主目录中存储其数据和配置。理解这种布局有助于备份、迁移和排查。

### 配置

主配置文件位于 `~/.openclaw/openclaw.json`，采用 JSON5 格式，这意味着你可以添加注释和后尾逗号——这些功能让手工编辑比严格的 JSON 更愉快。配置是分层的：环境变量覆盖配置文件的值，配置文件又覆盖内置默认值。这样你就能在环境变量中保留敏感令牌，同时保持文件中的静态配置。

### 会话状态与压缩

OpenClaw 将每次对话作为会话文件，存储在 `~/.openclaw/sessions/` 下，捕获该会话的对话历史，元数据和任何持久工具状态。

会话存储为仅附加的事件日志，支持分支功能，方便恢复状态、检查历史，并推理某回合在对话树中的位置。

为了保持模型上下文限制，OpenClaw 执行自动压缩：对较旧的对话部分进行摘要和持久化，以便会话继续而不丢失关键上下文。在压缩之前，系统可以运行一个轻量级的"内存刷新"步骤，促进持久信息存储到内存文件中；这有助于防止在旧回合被压缩时丢失重要细节，且当会话工作区不可写时会跳过这些细节。

### 记忆搜索

OpenClaw 维护可搜索的对话记忆，以便在你与代理互动时提供相关背景。当你提问时，系统会自动搜索过去对话中语义相似的讨论，并将这些上下文注入当前回合，这样代理就能引用你几周前谈过的内容，而不必重复。

内存系统通过 SQLite 数据库和向量嵌入存储数据 `~/.openclaw/memory/<agentId>.sqlite`。消息到达后，会自动被索引。该系统采用混合搜索，结合向量相似度（语义匹配）与 BM25 关键词相关性（精确令牌匹配）来寻找最相关的过去上下文。

**工作区中的内存文件**：
- **MEMORY.md** — 包含精心整理、稳定事实的长期记忆。该文件仅在私密/主会话中加载以保护隐私，绝不会在群聊中加载，以免他人看到你的个人上下文
- **memory/YYYY-MM-DD.md** — 每日笔记，提供每日活动的原始日志和上下文

**嵌入提供者选择**：
1. 如果已经配置了本地嵌入模型（`local.modelPath`），就用它
2. 否则，检查是否有 OpenAI API 密钥并使用 OpenAI 嵌入
3. 否则，检查是否有 Gemini API 密钥并使用 Gemini 嵌入
4. 如果没有，则禁用内存搜索

### 凭证

敏感认证数据会放在 `~/.openclaw/credentials/` 中。这包括频道认证令牌，如 WhatsApp 的会话数据，Discord 等平台的 OAuth 凭证，以及频道访问所需的其他秘密。文件权限限制在 0600（仅拥有者读写权限），并且该目录会自动排除在版本控制之外，以防止意外泄露。

## 安全架构

OpenClaw 通过多层安全实现纵深防御。每一层都提供不同类型的保护，它们协同作用，构建了全面的安全态势。

### 1. 网络安全

默认情况下，网关只绑定到 127.0.0.1，即你的环回接口。这意味着网关只能从本地机器访问，永远不会暴露在公共互联网中。远程访问需要通过以下几种支持的方法之一进行显式配置：

- **SSH 隧道**（推荐用于 VPS）：
  ```
  ssh -N -L 18789:127.0.0.1:18789 user@host
  ```

- **Tailscale Serve**（仅 tailnet HTTPS）：你的网关通过安全加密连接向 Tailscale 网络上的其他设备开放

- **Tailscale Funnel**：通过 Tailscale 的基础设施将你的网关暴露在公共互联网中

基于令牌或密码认证保护非回环绑定。

### 2. 认证与设备配对

基于设备的配对增加了额外的安全层。所有 WebSocket 客户端（控制 UI、节点、CLI 工具）在连接握手时都会包含设备身份。设备身份由设备 ID 和密码密钥组成。

当新设备连接时：
- 本地连接（环回或同一 Tailscale 网络）可配置为自动批准，以简化同主机工作流程
- 远程连接必须在握手过程中签署挑战声明，以证明其拥有有效凭证，并需明确批准

一旦批准，网关会为该设备发放一个设备令牌，允许后续连接而无须重新审批。这种基于设备的模型即使有人获得了你的身份验证令牌，也能防止未经授权的访问。

### 3. 频道访问控制

- **私信配对**：当 `dmPolicy="pairing"`（默认）时，未知发送方会触发特定流程
- **允许列表**：明确指定哪些电话号码或用户名可以与你的机器人互动
- **组策略**：包括提及要求和组特定的允许列表

### 4. 工具沙箱

OpenClaw 使用基于 Docker 的沙箱技术，按会话隔离工具执行。主会话（你作为的直接交互）通常会在主机上原生运行工具，并拥有完全访问权限。相比之下，DM 和组会话可以配置为在临时的 Docker 容器中执行工具，从而减少不受信任输入的影响。

每个沙箱容器都提供一个隔离的文件系统、可选的 network 访问（通常默认禁用，仅在需要时显式启用）以及可配置的资源限制（CPU/内存）。容器寿命短暂：它们是为沙盒执行而创建的，然后销毁，所以即使 DM 或组会话被强制采取不安全行为，"爆炸半径"也仅限于该容器，而非你的主机环境。

### 5. 即时注入防御

上下文隔离通过清晰区分输入，有助于防御即时注入攻击。用户消息包含源数据，系统指令与用户提供的保持独立，工具结果以结构化格式包装，以区别于用户输入。这种分离使攻击者更难欺骗代理将不可信消息视为系统指令。

## 部署架构

OpenClaw 支持四种主要部署模式，每种模式针对不同的用例和环境进行了优化。该架构在所有模式中保持一致；变化的是网关运行在哪里以及客户端如何连接到它。

### 1. 本地开发（macOS/Linux）

在本地开发环境中，所有操作都运行在你的开发机上。你在前景启动网关，使用 `pnpm dev`，这样可以启用代码更改时的热重载。网关绑定到 `127.0.0.1:18789`，仅能从本地机器访问。CLI 工具和网页界面直接连接到这个环回地址。不需要认证，因为环回接口被视为可信的，调试日志也以全速运行。

### 2. 生产版 macOS（菜单栏应用）

macOS 生产部署使用 LaunchAgent 作为后台服务运行网关。登录时服务会自动启动，并持续运行。macOS 菜单栏应用提供了启动、停止和重启网关的原生界面。它集成了直接嵌入应用中的 WebChat 界面、免提操作的语音唤醒功能，以及通过环回界面实现本地访问。远程访问可通过 SSH 隧道或 Tailscale 实现。

此部署支持原生 macOS 集成，包括 iMessage 支持，因为 iMessage 需要在 actual Mac 上运行。Voice Wake 与 ElevenLabs 集成，实现语音识别和合成。通过 A2UI 系统支持 Canvas，为代理驱动的界面提供了可视化工作区。

### 3. Linux/VM（远程网关）

在 VPS 或虚拟机上运行 OpenClaw，无需您的个人电脑保持开机，即可全天候 24 小时开放。网关作为系统服务运行在远程主机上，并可保持绑定为环回（127.0.0.1）以确保安全。你的本地客户端（CLI 和 Web UI）通过 SSH 隧道连接，该隧道将本地端口转发到远程环回端口。

- **选项 A：SSH 隧道（推荐默认）**
- **选项 B：Tailscale Serve**（仅 tailnet HTTPS）

### 4. Fly.io（容器部署）

Fly.io 是一种云原生部署选项，网关运行在由 Fly.io 管理的 Docker 容器中。持久卷存储 OpenClaw 状态（配置、会话、凭证），使其能够存活、部署和重启。Fly.io 在容器前方提供一个托管的 HTTPS 端点（带 TLS 终止），使网关可通过公共互联网远程访问。

因为网关在这种模式下可以从公共互联网访问，你应该启用强认证并将其视为面向互联网的服务。

## 结论

OpenClaw 代表了个人 AI 基础设施的现代化方法：本地优先、自托管且完全可控。其架构通过单进程网关模型实现简洁，同时支持多智能体路由、工具沙箱和可扩展插件的强大功能。这使得刚入门的开发者也能轻松上手，同时保持生产环境的准备，满足高需求。

围绕网关控制平面的枢纽辐射设计实现了消息平台间的统一访问。无论你是通过 WhatsApp、Discord 还是 iMessage 发消息，都能获得一致的客服体验。强有力的安全边界可以在不牺牲能力的前提下防止不受信任的输入。代理原生运行时，包含工具执行和持久会话，带来了真正的智能助手体验，而不仅仅是环绕大型语言模型的聊天包装器。

无论你是在笔记本上运行 OpenClaw 用于个人使用，还是部署到 VPS 上以实现全天候可用性，你都能从一个私密 AI 助手。你可以控制数据运行位置、如何曝光，以及数据如何存储和访问。

在 AI 能力日益被专有 API 和封闭环境锁住的时代，OpenClaw 提供了另一种选择：按自己的方式运行助手，通过你已有的渠道访问，并透明地了解其工作原理。
