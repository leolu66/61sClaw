## 1. 概述

### 1.1 背景

OpenClaw 系统需要自动化测试能力来验证：
- 技能功能是否正常
- Main Agent 响应是否符合预期
- 系统整体可用性

### 1.2 目标

- **自动化**: 一键执行测试套件
- **可扩展**: 支持新增测试用例
- **分层**: 支持不同级别的测试覆盖
- **端到端**: 真实模拟用户交互

### 1.3 设计原则

| 原则 | 说明 |
|------|------|
| 简单可靠 | 使用文件系统同步，无需额外服务 |
| 复用现有 | 利用 OpenClaw CLI 发送消息 |
| 灵活验证 | 关键词匹配，不强制精确匹配 |
| 自动清理 | 测试后自动清理临时文件 |

## 3. 测试模式

### 3.1 模式A：Skill Check（技能检查）

**用途**: 验证技能文件、数据是否存在，功能是否可用

**实现方式**:
```javascript
// 直接检查文件系统
fs.existsSync(skillPath)     // 检查技能文件
fs.readFileSync(dataPath)    // 读取数据文件
spawn(process)               // 启动进程验证
```

**适用场景**:
- 验证技能是否安装
- 检查配置文件
- 验证数据完整性
- 简单功能验证

**示例**:
```json
{
  "id": "260302-001",
  "type": "skill",
  "skill": "potplayer-music",
  "description": "验证播放音乐功能"
}
```

### 3.2 模式B：Test-Agent（端到端测试）

**用途**: 模拟真实用户，验证 Main Agent 的响应能力

**实现方式**:
```javascript
// 1. 发送测试消息
execSync('openclaw agent --message "【测试:ID】input"')

// 2. Main Agent 处理并响应

// 3. 记录并验证结果
test-hook.js 检测标记 → 验证响应 → 写入文件
```

**适用场景**:
- 验证对话能力
- 测试响应质量
- 验证意图识别
- 端到端流程测试

**示例**:
```json
{
  "id": "260302-006",
  "type": "agent",
  "input": "你好",
  "expected": "问候",
  "description": "发出你好指令测试"
}
```

### 3.3 模式对比

| 维度 | Skill Check | Test-Agent |
|------|-------------|------------|
| **测试对象** | 技能文件/数据 | Main Agent 响应 |
| **执行方式** | 本地检查 | 消息发送 |
| **验证内容** | 存在性/可用性 | 响应内容 |
| **复杂度** | 低 | 中 |
| **真实性** | 低 | 高 |
| **执行速度** | 快 | 较慢（需等待响应） |

## 5. 核心组件

### 5.1 组件清单

| 组件 | 路径 | 职责 | 输入 | 输出 |
|------|------|------|------|------|
| **test-runner** | `src/test-runner.js` | 主控模块，协调测试执行 | 测试级别 | 测试报告 |
| **test-case-manager** | `src/test-case-manager.js` | 管理测试用例 | 用例ID/配置 | 用例对象 |
| **real-runner** | `src/real-runner.js` | 执行具体测试 | 测试用例 | 测试结果 |
| **test-executor** | `../../workspace-test-agent/test-executor.js` | 发送测试消息 | 测试ID/输入 | pending文件 |
| **test-hook** | `../../test-hook.js` | 记录测试结果 | 消息/响应 | result文件 |
| **response-wrapper** | `../../response-wrapper.js` | 包装响应，自动记录 | 消息/响应 | 响应内容 |

### 5.2 组件详情

#### test-runner.js

**职责**:
- 接收用户测试指令
- 加载并筛选测试用例
- 调用 real-runner 执行测试
- 汇总并格式化测试结果

**关键方法**:
```javascript
runTest(level)      // 执行指定级别测试
formatResults()     // 格式化测试报告
```

#### real-runner.js

**职责**:
- 根据测试类型选择执行方式
- Skill Check: 检查文件/启动进程
- Test-Agent: 调用 test-executor
- 等待并读取测试结果

**关键方法**:
```javascript
executeTest(tc)              // 执行单个测试用例
_testWithAgent(tc)           // Test-Agent 测试
_waitForTestResult(id)       // 等待测试结果
_testSkill(skillName)        // Skill Check
```

#### test-executor.js

**职责**:
- 构建测试消息
- 使用 CLI 发送消息到 Main Agent
- 记录 pending 文件

**关键方法**:
```javascript
runTest(testCase)            // 发送测试消息
```

**CLI 命令**:
```bash
openclaw agent --session-id <id> --message "【测试:ID】input"
```

#### test-hook.js

**职责**:
- 检测消息中的测试标记
- 验证响应内容
- 写入 result 文件

**关键方法**:
```javascript
checkAndRecordTest(msg, response)  // 检查并记录
validateResponse(response, expected)  // 验证响应
isTestMessage(msg)                 // 检测测试标记
extractTestId(msg)                 // 提取测试ID
```

**测试标记格式**:
```
【测试:260302-006】你好
```

### 6.2 数据格式

#### 测试用例格式 (test-cases.json)

```json
{
  "testCases": [
    {
      "id": "260302-006",
      "createdAt": "2026-03-16",
      "level": 1,
      "description": "发出你好指令测试",
      "type": "agent",
      "input": "你好",
      "expected": "问候",
      "status": "enabled",
      "executionOrder": 1
    }
  ]
}
```

**字段说明**:
| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| id | string | 是 | 测试用例ID，格式：YYMMDD-XXX |
| createdAt | string | 是 | 创建日期 |
| level | number | 是 | 测试级别 1-4 |
| description | string | 是 | 测试描述 |
| type | string | 否 | 测试类型：skill/agent |
| input | string | 否 | Test-Agent 输入消息 |
| expected | string | 否 | 期望响应关键词 |
| skill | string | 否 | Skill Check 目标技能 |
| status | string | 是 | 状态：enabled/disabled |
| executionOrder | number | 否 | 执行顺序 |

#### Pending 文件格式

```json
{
  "testId": "260302-006",
  "input": "你好",
  "expected": "问候",
  "sentAt": "2026-03-16T15:30:00.000Z",
  "status": "pending"
}
```

#### Result 文件格式

```json
{
  "testId": "260302-006",
  "response": "你好！我是小天才...",
  "expected": "问候",
  "validation": {
    "success": true,
    "message": "包含预期关键词: 问候",
    "matched": true,
    "keyword": "问候"
  },
  "success": true,
  "recordedAt": "2026-03-16T15:30:05.000Z"
}
```

| T6 | Main Agent | 调用 test-hook | 消息+响应 |
| T7 | test-hook | 写入 result 文件 | 测试结果 |
| T8 | Auto-Tester | 读取 result 文件 | 通过/失败 |
| T9 | Auto-Tester | 汇总并输出报告 | 测试统计 |
| T10 | Auto-Tester | 清理临时文件 | 删除 pending/result |

### 9.3 验证规则

**关键词匹配**:
- 支持多个关键词，用逗号分隔
- 只要包含任一关键词即通过
- 不区分大小写
- 无 expected 时默认通过

**示例**:
```json
"expected": "天气,温度,北京"
// 响应包含"天气" → 通过
// 响应包含"温度" → 通过
// 响应包含"北京" → 通过
```

### 9.4 故障排查

| 问题 | 原因 | 解决 |
|------|------|------|
| 测试消息发送失败 | Gateway 未运行 | `openclaw gateway status` |
| 结果文件未生成 | test-hook 未调用 | 检查 Main Agent 是否集成 |
| 验证失败 | 关键词不匹配 | 检查 expected 和实际响应 |
| 超时 | Main Agent 未响应 | 检查系统状态 |

### 10.2 设计决策记录

| 决策 | 说明 | 原因 |
|------|------|------|
| 文件系统同步 | 使用 pending/result 文件 | 简单可靠，无需额外服务 |
| CLI 发送消息 | `openclaw agent --message` | 利用现有机制，无需新协议 |
| 关键词验证 | 检查响应是否包含关键词 | 灵活，不需要精确匹配 |
| 自动清理 | 读取结果后删除文件 | 避免文件堆积，保持整洁 |
| 30秒超时 | 默认等待结果时间 | 平衡响应速度和测试完整性 |
| 串行执行 | 一次执行一个测试 | 避免消息混乱，便于调试 |
