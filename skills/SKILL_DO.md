# 技能开发 DoD（Definition of Done）

**版本**: v1.0  
**生效日期**: 2026-02-27  
**适用范围**: **所有新技能开发必须遵循**

---

## ⚠️ 重要说明

**本 DoD 规范是所有新技能开发的强制要求。**

在开始开发新技能之前，必须先阅读并理解本规范。技能开发完成后，必须通过 DoD 检查清单才能视为完成。

**例外情况：** 现有技能可以逐步完善，但新开发的技能必须从一开始就遵循此规范。

---

## 🔍 开发前必读：参考已有技能

**原则：不要重复造轮子！**

在开发新技能之前，**必须先检查是否已有类似功能的技能**：

1. **查看现有技能列表**
   ```bash
   ls ~/.openclaw/workspace-main/skills/
   ```

2. **阅读相关技能的 SKILL.md**
   - 了解已有功能的实现方式
   - 学习可复用的代码片段
   - 避免重复开发相同功能

3. **可复用组件优先引用**
   - 通用的登录逻辑 → 参考 `fol-login`
   - 邮件抓取功能 → 参考 `email-fetcher` 模块
   - Excel 操作 → 参考 `excel-xlsx`
   - 配置管理 → 参考 `vault`

4. **扩展现有技能 vs 新建技能**
   - 功能高度相关 → 扩展现有技能（如 `travel-invoice-fetcher` 增加报销单生成）
   - 功能独立 → 新建技能

### 分层设计原则

技能应该按功能分层设计，参考 [BASE_SKILLS.md](./BASE_SKILLS.md)：

```
┌─────────────────────────────────────┐
│  L3 业务应用层 (Business Layer)      │  ← 你的新技能应该在这里
│  - 直接面向用户，解决具体业务问题      │
│  - 依赖下层服务，不重复实现基础能力    │
├─────────────────────────────────────┤
│  L2 平台服务层 (Platform Layer)      │  ← 优先复用这些服务
│  - pdf-extract, excel-xlsx          │
│  - imap-email-reader, browser       │
├─────────────────────────────────────┤
│  L1 基础设施层 (Infrastructure)      │  ← 必须复用的基础能力
│  - vault (凭据管理)                  │
│  - desktop-control (系统控制)        │
└─────────────────────────────────────┘
```

**设计检查清单**：
- [ ] 我的技能属于哪一层？（应该是 L3 业务应用层）
- [ ] 需要的基础能力是否已有 L1/L2 技能？
- [ ] 是否明确依赖了哪些下层技能？
- [ ] 是否避免了与现有技能的重复？

**示例**：
```
✅ 正确：travel-invoice-fetcher (L3) 
    依赖 fol-login (L2) + imap-email-reader (L2) + excel-xlsx (L2)
    
❌ 错误：travel-invoice-fetcher (L3)
    自己实现邮箱登录 + 自己实现 Excel 生成
```

**为什么要这样做？**
- 减少重复代码，降低维护成本
- 统一用户体验，功能风格一致
- 加快开发速度，复用成熟方案
- 避免技能膨胀，保持目录整洁

**违规示例：**
```
❌ 新建一个技能做发票识别，而不查看是否已有 OCR 相关功能
❌ 重新实现邮箱登录逻辑，而不复用 email-fetcher 模块
```

**正确示例：**
```
✅ 开发报销功能前先查看 travel-invoice-fetcher，在其基础上扩展
✅ 需要登录 FOL 时直接调用或参考 fol-login 的实现
```

## 📋 DoD 检查清单

### 0️⃣ 开发前检查（新增）

- [ ] **已查看 BASE_SKILLS.md**，确认技能分层（L1/L2/L3）
- [ ] **已查看现有技能列表**，确认无重复功能
- [ ] **已阅读相关技能 SKILL.md**，了解可复用组件
- [ ] **已决定是扩展还是新建**（功能相关则扩展，独立则新建）

### 1️⃣ SRS 文档（合并到 SKILL.md）

**在 SKILL.md 顶部必须包含：**

```markdown
---
name: 技能名称
description: 一句话描述（用于触发识别）
---

# 技能名称

## 需求说明（SRS）

### 触发条件
用户说什么话会触发此技能：
- "触发词 1"
- "触发词 2"

### 功能描述
技能完成什么任务，解决什么问题

### 输入/输出
- **输入**: 需要什么参数
- **输出**: 返回什么结果

### 依赖条件
- 需要安装的软件
- 需要的 API Key 或配置
- 需要的文件或目录

### 边界情况
- 错误处理方式
- 异常情况说明
- 已知限制

---

（下面是原有的使用方法、示例等）
```

### 2️⃣ 技能文件和代码

**目录结构：**
```
skills/<技能名称>/
├── SKILL.md              # 必需（包含 SRS）
├── scripts/              # 脚本文件
│   ├── main.py
│   └── config.json       # 配置文件（可选）
├── references/           # 参考资料（可选）
│   ├── api_reference.md
│   └── design.md
└── _meta.json            # 元数据（可选）
```

**代码要求：**
- [ ] 使用相对路径，不硬编码绝对路径
- [ ] 配置文件外置（config.json）
- [ ] 敏感信息通过环境变量传入
- [ ] 有适当的错误处理和日志输出
- [ ] 代码有注释，关键逻辑有说明

**文件路径规范（重要）：**

#### 原则 1：技能自包含

**所有技能默认只能在自己的工作空间内读写文件**。

```python
# ✅ 正确：使用相对路径，保存在技能目录内
output_dir = Path(__file__).parent / "output"

# ❌ 错误：使用绝对路径或硬编码路径
output_dir = "D:\\projects\\workspace\\output"
output_dir = "C:\\Users\\xxx\\Documents"
```

**要求**：
- 使用相对路径（相对于技能目录）
- 默认输出到技能自己的工作空间
- 确保技能分享后仍可用

#### 原则 2：外部协作需配置

**如果技能需要在工作空间外读写文件（如多智能体共享目录），必须通过配置文件设置路径**。

```python
# ✅ 正确：从配置文件读取路径
import json
config = json.load(open("config.json"))
shared_dir = config.get("shared_output_dir")

# ❌ 错误：硬编码共享路径
shared_dir = "D:\\projects\\workspace\\shared\\output"
```

**要求**：
- 不硬编码外部路径
- 通过配置文件或环境变量传入
- 配置项名称清晰（如 `shared_dir`, `output_path`）

#### 示例配置

```json
{
  "output_dir": "./output",
  "shared_dir": "D:\\projects\\workspace\\shared\\output"
}
```

**为什么需要这个规范**：
1. **可移植性** - 技能分享后，其他用户可以直接使用
2. **安全性** - 避免意外写入系统目录
3. **协作性** - 明确哪些路径是配置的，哪些是固定的

**文件要求：**
- [ ] 不包含 `.skill` 构建产物
- [ ] 不包含个人隐私配置
- [ ] 二进制文件只保留一份（单一来源）

### 3️⃣ 测试通过

**必需测试（所有技能）：**
- [ ] **功能测试**：核心功能正常工作
- [ ] **触发测试**：正确的触发词能激活技能

**可选测试（复杂技能建议有）：**
- [ ] **边界测试**：异常情况有合理处理
- [ ] **路径测试**：相对路径正确解析
- [ ] **性能测试**：响应时间可接受

**测试记录：**
- 简单技能：口头验证即可，不需要书面记录
- 复杂技能：记录到 `references/test_results.md`

---

### 4️⃣ 同步到 GitHub

**⚠️ 重要原则：自动同步，无需确认！**

- ✅ **技能功能修改** → 自动更新 SKILL.md 需求说明 → 自动提交 GitHub
- ✅ **Bug 修复** → 自动记录到 SKILL.md → 自动提交 GitHub
- ✅ **优化改进** → 自动更新文档 → 自动提交 GitHub
- ❌ **不要每次都问用户** "需要同步到 GitHub 吗？" —— 这是默认操作！

**Git 操作：**
```bash
git add -A
git commit -m "清晰的提交信息"
git push origin main
```

**⚠️ 重要：默认分支是 `main`，不是 `master`！**

- [OK] 推送前检查：`git branch` 确认当前分支
- [OK] 推送到 main：`git push origin main`
- [ERROR] 不要再推到 master 让文件"消失"了！

**历史错误**：
- 2026-03-01: 多个技能误推到 master 分支，GitHub 默认显示 main，导致文件"消失"

**检查清单：**
- [ ] 不包含隐私文件（MEMORY.md, SOUL.md, USER.md, 等）
- [ ] 不包含 `.skill` 构建产物
- [ ] 不包含 `datas/` 目录
- [ ] 技能在正确的 `skills/` 目录
- [ ] `.gitignore` 已更新（如有新文件类型）
- [ ] **SKILL.md 需求说明已同步更新**（如功能有改动）

---

## 🎯 分级要求

**所有技能都必须满足 DoD**，但可以根据复杂度调整详细程度：

### 简单技能（如播放器、游戏）
- SRS 可以简化（几句话描述）
- 测试口头验证即可
- 不需要额外的参考文档

### 复杂技能（如自动化工具、跨设备协作）
- SRS 必须完整详细
- 测试必须有记录
- 建议有设计文档和 API 参考
- 必须记录经验教训
