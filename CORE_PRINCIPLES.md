## 海恩法则（Heinrich's Law）

### 核心内容
每项重大技术事故背后，必然有 29 潜微事故、300 次未遂先兆以及 1000+ 潜微隐患。

### 核心定义
隐患常积于忽微，溃之毫厘，谬以千里。中文谚语：祸患常积于忽微，溃之毫厘。

### 关键数字
- 29 潜微事故
- 300 次未遂先兆
- 1000+ 潜微隐患

### 三个关键洞察
1. **小问题积少成大事故** - 不重视小问题，累积成大灾难
2. **轻视隐患导致灾难** - 1000+ 个小隐患，累积成无法挽回的灾难
3. **未遂先兆被忽略** - 300 次小预警，最终导致大事故

## 隐私保护原则

> 任何日志、记忆、文档中不得记录明文敏感信息

**敏感信息范围**：
- 身份证号、护照号等证件号码
- 手机号、固定电话号码
- 密码、API Key、Token 等认证凭据
- 银行卡号、支付宝账号等财务信息
- 家庭住址、邮箱等个人联系方式

**脱敏规则**：
- 身份证号：保留前 6 位和后 4 位，中间用 `*` 替换，如 `320106********3657`
- 手机号：保留前 3 位和后 4 位，中间用 `*` 替换，如 `138****5678`
- 密码/密钥：绝不记录，必须存储在加密的 vault 中

**存储原则**：
- ✅ 敏感信息 → 使用 `vault` 技能加密存储
- ❌ 明文敏感信息 → 禁止出现在日志、记忆、文档、代码中

## 策略原则

### 让子弹飞一会
> 一件新事务出现时，不急着用旧经验去限制或干扰，让它先自由发展一段时间，等看清楚发展趋势或将出现严重问题前再去干涉。

**核心要点：**
- **不急于评判** - 避免用过去的经验框架过早定义新事物
- **观察优先** - 给事物自然展现的空间和时间
- **顺势而为** - 等趋势明朗或问题浮现再介入
- **降低试错成本** - 避免因过早干预导致的方向错误

**适用场景：**
- 新技术/新工具评估（如 OpenClaw、Claude Code）
- 技术选型对比（框架 A vs 框架 B）
- 架构方案选择
- 是否进行大规模重构
- 新功能是否值得投入开发
- 多方案并行验证效果
- 团队/人员的成长和表现观察

**不适用场景：**
- ❌ 已知问题/技术债务（应小步快跑，逐步修复）
- ❌ 安全漏洞（应立即修复）
- ❌ 影响稳定性的 bug（应优先修复）
- ❌ 上游已明确弃用/废弃的依赖（应逐步更新）

## 小步快跑，逐步迭代

> 对已知问题和技术债务，不要拖延或等待，应该快速行动、小步前进、持续迭代，一个个解决问题，避免累积成大问题。

**核心要点：**
- **及时响应** - 已知问题不拖延，避免技术债务累积
- **小步前进** - 每次只改一点，降低风险和回滚成本
- **持续迭代** - 逐步完善，不求一次性完美解决
- **优先级排序** - 先解决影响大、风险高的问题

**适用场景：**
- 依赖包更新/升级（如 npm deprecated 警告）
- 代码重构和优化
- Bug 修复和维护
- 性能问题
- 技术债务清理

**与"让子弹飞一会"的区别：**
- ✅ "小步快跑" → 已知问题，方向明确，需要行动
- 🚫 "让子弹飞一会" → 新事物，方向不明，需要观察

## 时机不到，不要勉强，顺势而为

> 当做一个事情所依赖的基础设施不完善，或技术不成熟时，不要强求最佳效果，学会在成本和效果之间做平衡和权衡。

**核心要点：**
- **识别约束** - 判断哪些是外部约束（基础设施、技术成熟度），哪些是内部约束（时间、人力）
- **成本效益平衡** - 不追求完美方案，在现有条件下找最优解
- **分阶段投入** - 早期用简单方案验证，等时机成熟再优化
- **顺势而为** - 跟随技术生态自然演进，不逆流而上

**适用场景：**
- 新技术早期采用（生态不完善时）
- 依赖外部平台/基础设施的功能
- 团队技能成长中（早期不强行上高难度技术）
- 跨团队协作依赖外部进度

**问题报告策略：**
- 🔴 高优先级 - 小众但影响严重（只有你知道，立即报告）
- 🟡 中优先级 - 通用但影响严重（先搜索是否已有人报告）
- 🟢 低优先级 - 警告/提示类（不报告，等官方自然修复）
- ⚪ 忽略 - 常规提示，无需报告

**与相关原则的关系：**
- 🎯 "让子弹飞一会" → 新事物，方向不明，先观察
- 🏃 "小步快跑" → 已知问题，快速迭代
- ⚖️ "时机不到" → 外部约束，成本效益权衡

## 工程实践原则

### 分层设计原则

> 下层能力要简单、可靠、独立，复杂的功能在上层再组装，增加灵活性。

**核心要点：**
- **底层简单可靠** - 每个底层模块只做一件事，做到极致稳定
- **上层组装灵活** - 复杂功能通过组合底层模块实现
- **保持独立性** - 底层不依赖上层，便于替换和复用
- **分层解耦** - 改变一层不影响其他层

**典型架构：**
- 数据层：文件读写、数据库操作
- 业务逻辑层：数据处理、验证、计算
- 展示层：UI 渲染、图表生成
- 接口层：API 调用、消息推送

## 领域模型构建原则

> 要在交互中，逐步构建总体的视图和模型，在这个模型基础上去设计程序会更灵活。

**核心要点：**
- **先有模型，再写代码** - 在实践中形成对领域的认知，构建模型
- **总体视图先行** - 先画大图，了解全貌，再切分模块
- **路径/分类/策略** - 每个功能形成清晰的路径、分类、策略
- **便于迭代** - 模型清晰后，后期维护和扩展都更方便

**示例（几何万国游戏）：**
- 总体视图：资源、任务、地图、成就
- 分层模型：底层操作 → 中层策略 → 高层决策
- 每个功能路径：领取奖励 → 检查条件 → 执行领取 → 记录状态
- 分类清晰：日常任务 / 随机事件 / 长期目标
- 策略分明：什么时候做什么最优化

**记录时间：** 2026-02-21

## 复杂任务分解原则

> 面对复杂任务时，先进行任务分解，将大任务拆分为相对独立的子功能。

**执行步骤：**
1. **识别核心功能** - 先关注核心功能，再完善细节和辅助功能
2. **模块分解** - 保持每个模块的高内聚、低耦合
3. **简单原型先行** - 先从简单原型和技术验证开始，实现关键技术点的 0→1 突破
4. **逐步扩展** - 核心功能验证后再考虑 1→N 的拓展

**示例：**
- 微信公众号文章获取工具分解为：列表获取 → 内容读取 → 文件保存 → 报告生成
- 每个模块独立，通过接口交互
- 便于并行开发和测试

## 外部约束关注原则

> 实现功能时要关注外部资源的约束和限制，避免被禁止或限制。

**反爬策略：**
1. **随机延迟** - 请求间添加随机延迟，避免固定频率
2. **随机顺序** - 处理顺序随机化，避免可预测模式
3. **控制并发** - 避免高密度、大批量并发请求
4. **模拟真实行为** - 使用浏览器模拟、随机 User-Agent

**示例：**
- 微信公众号文章读取：2.5 秒随机延迟、随机获取顺序、单线程处理
- 避免触发反爬：速率限制、User-Agent 检测、行为模式识别

**记录时间：** 2026-02-18

## 可维护性设计原则

> 把可变和不变的内容隔离开来，避免每次修改都要大面积改程序。

**实践方法：**
1. **配置外置** - 将可变参数（公众号列表、路径、阈值等）放到外部配置文件
2. **环境变量** - 敏感信息（密钥、Token 等）通过环境变量传入
3. **命名模板** - 保持可配置的命名格式，如 `{account}_{title}.md`
4. **参数化设计** - 延迟时间、超时时间、数量限制等都做成可配置参数

**示例：**
- `config.json` 管理公众号列表，添加新号无需改代码
- 环境变量 `WECHAT_COOKIE` 传入敏感信息
- 命名模板 `account_{date}.md` 便于批量处理

## 错误学习方法论

### 错误是最好的老师

> 每次遇到错误时，需要去反思过程，建立错误日志，定期从历史错误中检索和学习。

**核心要点：**
- **错误即学习** - 每次错误都是成长的机会，不要逃避或忽略
- **过程反思** - 不仅修复错误，还要理解为什么出错
- **错误日志** - 记录错误类型、原因、解决方案，形成知识库
- **定期回顾** - 从历史错误中检索，避免重复犯错
- **举一反三** - 一个错误的教训可以应用到类似场景

**执行方式：**
1. 遇到错误时，先记录错误现象和上下文
2. 分析根本原因，不只是表象
3. 总结解决方案和预防措施
4. 定期回顾错误日志，强化记忆
5. 将错误经验应用到新任务中

## Git 仓库管理规范

### 技能目录结构原则

> 所有技能统一在 `skills/` 目录，`agfiles/` 只包含工具脚本。

**目录规范：**
- `skills/<技能名称>/` - 完整技能（包含 SKILL.md 和 scripts/）
- `agfiles/` - 工具脚本（.py, .ps1, .bat），不包含 SKILL.md
- `datas/` - 本地数据和状态文件（不上传 GitHub）

**排除规则（.gitignore）：**
```
# 技能构建产物（自动生成，不纳入版本管理）
*.skill

# 个人数据和状态文件（不同设备不同，不上传 GitHub）
SOUL.md
IDENTITY.md
USER.md
MEMORY.md
TOOLS.md
AGENTS.md
BOOTSTRAP.md
HEARTBEAT.md
CORE_PRINCIPLES.md

# 工作日志（本地记录，不同设备不同）
logs/

# 过程性文件（分析报告、图表等）
agfiles/
```

**理由：**
- `.skill` 是 ZIP 打包格式，是构建产物
- 应该管理源代码（skills/ 目录）
- 类似于不将编译后的 .exe 纳入版本管理

**记录时间：** 2026-02-27

### 单一来源原则

> 文件只保留一份，避免冗余和同步问题。

**错误示例：**
- ❌ gobang.html 复制两份（根目录 + 技能目录）
- ❌ 同一技能的多个版本并存

**正确做法：**
- ✅ 技能文件随技能走（如 `skills/gobang-game/gobang.html`）
- ✅ 使用相对路径查找，不硬编码绝对路径

**记录时间：** 2026-02-27

### 自动同步原则

> 本地技能和 agfiles 的修订自动同步 GitHub，不需要确认。

**工作流：**
1. 修改技能或 agfiles 文件
2. `git add -A; git commit -m "描述"`
3. `git push origin master; git push origin master:main`
4. 完成，不需要询问确认

**例外情况（需要确认）：**
- 重大架构调整
- 删除重要文件或目录
- 涉及隐私或安全的变更

## 技能开发规范

### 技能开发 DoD（Definition of Done）

> 所有技能开发都必须满足 DoD 要求，确保质量和一致性。

**DoD 四要素：**
1. **SKILL 文档** - 整合到 SKILL.md 中（触发条件、功能说明、依赖安装、使用方法）
2. **技能文件和代码** - 规范目录结构，使用相对路径，配置外置
3. **测试通过** - 功能测试 + 集成测试（简单技能口头验证，复杂技能书面记录）
4. **同步到 GitHub** - 无隐私文件，无 .skill 构建产物，推送到 master 和 main

**文件位置：** `skills/SKILL_DO.md`（完整 DoD 规范）

**分级要求：**
- 简单技能：SKILL 简化，测试口头验证
- 复杂技能：SKILL 完整，测试书面记录，建议有设计文档
- 自动化脚本：添加 README 和使用说明

## 时间线分析报告

### 核心发现（基于 2026-02-10 至 02-27 数据）

#### 1. 用户习惯演变四大阶段

| 阶段 | 日期 | 使用模式 | 主要特征 | 典型任务 |
|------|------|---------|---------|---------|
| **观察期** | 02-10 至 02-15 | 均衡型 | IO 比 0.08 | 2 种模型 | 日常维护、文档处理 |
| **过渡期** | 02-16 至 02-20 | 均衡型 | IO 比 0.09 | 单一模型 | 中等开发、深度分析 |
| **阅读期** | 02-21 至 02-25 | **阅读型** | IO 比 0.004 | MiniMax 主导（95%） | 代码审查、文档分析、知识检索 |
| **生成爆发期** | 02-26 至 02-27 | **大型生成** | IO 比 0.028 | 混合模型 | 代码生成、长文本输出、批量任务 |

#### 2. 模型选择优化策略（基于 IO 比）

| IO 比范围 | 推荐模型 | 限制模型 | 理由 |
|-----------|---------|---------|------|
| < 0.01 | QWen-Plus | Doubao-Pro | 阅读任务，性价比最优 |
| 0.01-0.05 | MiniMax-M2.5 | - | 均衡任务，中等复杂度 |
| 0.05-0.15 | MiniMax-M2.5 | - | 生成任务，长文本能力 |
| > 0.15 | MiniMax-M2.5 | Doubao-Pro | 大型生成，强制思考 |

**核心洞察：**
- MiniMax 在阅读模式下单位成本高（¥2.3），但在生成模式下效率极高（¥0.007）
- QWen-Plus 性价比最高（¥0.037），在阅读/均衡型任务中应优先使用
- Doubao-Pro 单位成本极高（¥10.301），应限制在极复杂推理任务
